# Bundled Desktop App PRD

## Product name
Bundled Ryan Report Desktop App

## Purpose
This product is a locally installed desktop application for macOS and Windows that allows a client to run the current Ryan report workflow on their own machine with the same practical behavior as the current app, but with a more controlled and more reliable automation environment.

The product must not require customer hosting, Docker, server deployment, browser setup, Python installation, or command-line use.

## Product goal
The goal is to preserve the current user outcome and current local workflow while removing avoidable setup and environment fragility.

The client should be able to install the app on a local Mac or Windows machine, complete first-run setup, click Run, and receive the same report outputs they receive today.

## Core product decision
This product is a bundled desktop application, not a hosted app and not a customer-facing containerized product.

The application must ship with and control its own automation browser/runtime instead of depending on the client’s separately installed browser, browser executable path, or browser profile configuration.

## In scope
- macOS desktop application installer/package.
- Windows desktop application installer/package.
- First-run local setup wizard.
- Local credential storage.
- Local scheduling on the client machine using OS-native scheduling behavior equivalent to what exists now.
- Bundled automation browser/runtime managed by the app.
- Local download of source reports.
- Local report build process and output generation.
- Local storage of state, configuration, logs, and output history.
- Configurable automation paths for reports so a client can create or update the click path for a report if the source website navigation changes.

## Out of scope
- Airtable push or any Airtable configuration/UI.
- Customer hosting.
- Customer use of Docker or containers.
- Manual upload fallback.
- Browser selection by the client.
- Reliance on the client’s installed Chrome, Edge, Brave, or other browser.
- Multi-tenant cloud administration.

## Users

### Primary user
A non-technical or lightly technical client user who wants to install the app, sign in once, and run the report locally with minimal setup.

### Secondary user
An internal support or admin user who may need to modify a report navigation path if Axon changes the menu structure.

## Current-state behavior to preserve
The current app is a local desktop application that runs a report pipeline in the background, validates configuration, downloads reports from Axon through browser automation, builds the Ryan report, writes local outputs, appends to XLSX, and supports macOS and Windows scheduling.

The new product must preserve those outcomes so that from the client’s perspective the app still “works as it does now,” meaning local install, local run, local outputs, and repeatable scheduled execution.

## Required product outcomes
The product must allow a client to:
- Install on macOS or Windows.
- Launch a desktop app normally from the OS.
- Enter and save credentials locally.
- Run the workflow manually with one primary action.
- Schedule the workflow locally.
- Generate the same report outputs currently generated by the existing app.
- Continue to maintain the historical Ryan file behavior used today.
- Modify the automation path for a report when the source website navigation changes.

## Functional requirements

### Installation
The client must receive:
- A macOS installer/package that installs the app locally.
- A Windows installer/package that installs the app locally.

The installer must include:
- The desktop application shell.
- The execution runtime.
- The report build logic.
- The automation browser/runtime.
- Default report definitions.
- Default settings template.

The client must not need to install Python, Playwright, Node, Chrome, Edge, or any extra dependency separately.

### First-run setup
On first launch, the app must guide the client through a setup flow that asks only for information required to perform the workflow.

Required setup inputs:
- Axon base URL.
- Username.
- Password.
- Local output/download directory.
- Historical Ryan file location, if applicable to current workflow behavior.

The setup flow must validate values before completion and provide plain-English fixes when required information is missing or invalid.

### Automation browser model
The app must launch and manage its own bundled automation browser/runtime for all report download activity.

The client must not be asked to:
- choose a browser,
- enter a browser executable path,
- choose a profile directory,
- manage CDP settings,
- manage browser user-data directories.

The automation browser must use an app-owned local profile stored in the application data area for the current user, separate from the client’s personal browser profile.

### Manual run
The main screen must expose one primary action: Run Report.

When selected, the app must:
1. Validate required settings and local paths.
2. Launch the bundled automation browser/runtime.
3. Log into Axon using saved credentials, handling verification code flow if required by the website.
4. Execute the configured report download paths.
5. Save downloaded source files to the configured local directory.
6. Run the local report build process using the downloaded files and the historical Ryan file.
7. Write the final output files locally.
8. Append new rows to the XLSX output in the same practical way as the current app.
9. Display success, error, and output status in the app UI.

### Scheduled run
The client must be able to enable and manage scheduled local runs within the app UI.

On macOS, the app must use native OS scheduling behavior equivalent to LaunchAgents-style scheduling.
On Windows, the app must use native OS scheduling behavior equivalent to Task Scheduler-based scheduling.

The scheduled run must execute the same workflow as a manual run using the same saved local settings.

### Outputs
The app must continue to create the same categories of local outputs as the current product:
- generated report CSV output,
- append output,
- maintained historical Ryan file behavior,
- XLSX append/create behavior where applicable.

All outputs must remain local on the client machine.

### Logs and troubleshooting
The app must provide a local log view that shows high-level run progress and readable errors during validation, login, download, build, and finalization.

The app must store local logs for troubleshooting purposes.

Errors must be rewritten into plain language for the client. Technical details may be retained in a secondary advanced log view.

### Report path configurability
The only end-user configurable automation behavior that must be exposed is report path management.

A report path is the ordered navigation path the automation uses to reach a report export action on the source website. The current system already models reports as ordered click steps such as tab, menu, submenu, and export button actions.

The new app must allow an authorized local user to:
- view existing report paths,
- enable or disable a report path,
- create a new report path,
- edit an existing report path,
- remove an obsolete report path.

### Report path editor requirements
The report path editor must support a path definition composed of ordered steps, each step containing:
- step type,
- display label,
- target text/name,
- wait time,
- whether the step triggers a download,
- optional timeout for download steps.

Supported step types in v1:
- Click tab
- Click menu item
- Click button

The user experience for path creation must be simplified compared with raw JSON editing.

The UI must provide:
- a structured form editor for steps,
- reorder controls,
- duplicate step control,
- add/remove step actions,
- a test-run option for a single report path,
- save and version the path locally.

The product must not require the client to edit JSON directly even if JSON remains the internal storage format.

### Default reports
The app must ship with default report definitions equivalent to the current default report set, including the currently configured report paths that are enabled by default.

### Historical Ryan handling
The app must preserve the current historical Ryan behavior:
- use a configured historical Ryan file when present,
- write new-only outputs,
- append outputs appropriately,
- create or update XLSX output locally as the current process does.

The app may improve the UX around historical file setup, but it must preserve practical behavior and output expectations.

## Non-functional requirements

### Reliability
The app must reduce local-environment fragility by bundling and controlling the automation browser/runtime instead of relying on user-installed browser paths and profiles.

### Cross-platform consistency
The app must provide the same functional workflow on macOS and Windows, with only OS-specific packaging, file paths, and scheduler implementation differing under the hood.

### Simplicity
The client-facing UI must be minimal and plain.

Preferred top-level sections:
- Run
- Settings
- Schedule
- Reports
- Logs

### Security
Credentials must be stored locally using OS-appropriate secure local storage where feasible. If not feasible in v1, credentials must at minimum be stored in an application-owned local configuration area with clear upgrade path to secure storage.

### Performance
A manual run should begin within a few seconds of launch and should provide immediate visible status updates while the background process runs.

## Replacement plan

### What is replaced
The following current concepts must be removed from the customer-facing setup model:
- customer browser selection,
- customer browser executable path,
- customer browser user-data directory,
- customer browser profile selection,
- Airtable configuration and push behavior.

### What stays
The following product characteristics stay:
- desktop application delivery model,
- local installation,
- local scheduling,
- local outputs,
- local historical file behavior,
- local report build logic,
- report-path based automation model,
- configurable report definitions/path editing.

### What changes
The major behavior change is that automation uses a bundled app-managed browser/runtime rather than a customer-managed browser environment.

## User experience flow

### First launch
1. Client installs app.
2. Client opens app.
3. App shows setup wizard.
4. Client enters Axon URL, username, password, output folder, and historical Ryan file location if needed.
5. App validates settings.
6. App saves setup.
7. App lands on Run screen.

### Manual run flow
1. Client clicks Run Report.
2. App validates settings and required paths.
3. App launches bundled browser/runtime.
4. App logs into Axon.
5. App runs each enabled report path.
6. App downloads source files locally.
7. App runs local build pipeline.
8. App writes final output files locally.
9. App updates status and logs.

### Report path update flow
1. Client opens Reports.
2. Client selects an existing report or chooses Create New Report Path.
3. Client defines the ordered path steps in the editor.
4. Client test-runs the path.
5. Client saves the path.
6. Path becomes available for future runs.

## Data model requirements
The application configuration must include:
- auth settings,
- download/output directory settings,
- historical Ryan file reference,
- schedule settings,
- report definitions and path steps,
- application state and logs.

The configuration must not include customer browser executable path or customer browser user-data path in the final bundled product configuration model.

## Packaging requirements

### macOS
The app must be distributed as a macOS installable desktop package/app bundle appropriate for a standard end user.

### Windows
The app must be distributed as a Windows installable desktop application for a standard end user.

Both packages must include the bundled automation runtime and all required internal dependencies.

## Acceptance criteria
The product is accepted when the following are true:

1. A client can install the application on macOS without separately installing Python or a browser for automation.
2. A client can install the application on Windows without separately installing Python or a browser for automation.
3. The app can run locally and produce the same practical report outputs as the current app.
4. The app no longer requires browser executable path, browser user-data directory, or browser profile configuration from the client.
5. The app supports local scheduling on macOS and Windows.
6. Airtable functionality is absent from the product.
7. A client can create, edit, save, and test a report path when a website navigation path changes.
8. The app stores outputs locally and preserves current local workflow expectations.
9. The app preserves historical Ryan and XLSX append behavior at a user-visible level equivalent to the current process.
10. The client can use the app with one primary run action and without command-line interaction.

## Open implementation note
If the source website offers a stable API or export endpoint in the future, the automation layer may later be replaced without changing the client-facing desktop product model. Until then, the application will continue to use browser automation, but in an app-owned bundled browser environment rather than the client’s browser environment.
